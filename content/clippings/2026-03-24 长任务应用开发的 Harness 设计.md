## 概览

本文探讨 harness 设计——围绕 AI Agent 行为的脚手架——如何让 Claude 产出更高质量的前端设计、并自主构建完整应用。

## 核心问题

作者识别了**两个持久挑战**：

1. **上下文窗口退化**：模型在长任务中随着 context 填满，连贯性下降。Sonnet 4.5 表现出"**context anxiety**"——在接近感知上限时过早草草收尾。

2. **自评偏差**：当 Agent 给自己的产出打分时，会**一贯地夸赞平庸作品**，对设计这类主观任务尤其如此。

## 解决方案：GAN 启发的架构

作者借鉴**生成对抗网络（GAN）** 的 generator-evaluator 模式来解决这些问题。

### 前端设计实验

系统用四个评分维度：

- **设计质量**：颜色、字体、布局组合出的连贯视觉身份
- **原创性**：是否有自定义决策而非通用模板
- **工艺（craft）**：技术执行（层级、间距、色彩和谐）
- **功能性**：可用性与任务完成度

> "评分标准明确惩罚高度通用的 **'AI slop'** 模式"——逼模型敢于做审美冒险。

Evaluator 通过 **Playwright MCP** 直接访问活页面，截屏 + 批评。**经过 5–15 轮迭代**，设计有明显改善，之后趋于平台期。

### 全栈开发 Harness

针对自主编码，作者搭了**三 Agent 系统**：

**Planner Agent**：把 1–4 句话的简单 prompt 扩展成详细产品规格，**刻意回避粒度过细的技术细节**，防止级联错误。

**Generator Agent**：用 React、Vite、FastAPI、SQLite/PostgreSQL 实现功能，内建 git 版本控制管理复杂度。

**Evaluator Agent**：通过 Playwright 测试运行中的应用——点击界面找 bug、验证数据库状态、按硬阈值打分。

## 关键创新：Sprint 合约

实现之前，generator 和 evaluator 谈判出一份"**sprint contract**"——关于可交付物和成功标准的约定。这弥合了"高层用户故事"和"可测试实现"之间的鸿沟，**而不会过早过度规范**。

通信通过**文件 handoff** 进行，保持 Agent 之间清晰的上下文边界。

## 性能对比

| 方案 | 时长 | 成本 | 结果 |
|---|---|---|---|
| **单 Agent** | 20 分钟 | $9 | 核心功能坏、流程僵化、布局差 |
| **完整 Harness + Opus 4.5** | 6 小时 | $200 | 抛光界面、可玩机制、集成 AI |

Harness 的额外开销换来了**真正有功能的应用**，而不只是表面光鲜。

## 随 Opus 4.6 的演进

模型升级后，作者**系统性地删除 harness 组件**：

- **删除**：Sprint 分解（模型能维持更长的连贯工作）
- **保留**：Planner（防止 under-scoping）
- **修改**：Evaluator 从"每 sprint"移到"end-of-build QA"

Evaluator 的价值变得**有条件**——在模型挣扎的任务边界处必要，在基线能力以内的任务上不必要。

## DAW 案例

数字音频工作站（DAW）生成示例展示了升级后的 harness 效率：
- **总时长**：3 小时 50 分
- **总成本**：$124.70
- **拆解**：4.7 分钟规划、3+ 小时构建、若干轮 evaluator

QA Agent 识别出关键缺口：时间线 clip 的交互拖拽、乐器 UI 面板、视觉效果编辑器尚未实现。Evaluator 的反馈触发了三轮修复。

## 核心发现

作者强调，**Evaluator 反馈捕获的是实质性问题**，不是表面 bug。对于 DAW，初始 QA 写道：

> "虽然应用看起来很惊艳，AI 集成也运作良好，**但若干核心 DAW 功能是 display-only 的，缺乏交互深度**。"

**模型改进并不消除 harness 的必要性**，而是把它的应用转向更宏大的目标——脚手架让能力超越基线性能。

## 更广泛的原则

1. **分解很重要**：把复杂任务拆成专门化 Agent 角色，防止连贯性丢失
2. **结构化 handoff**：跨会话携带状态的 artifact 让 context reset 成为可能
3. **外部评估**：把生成和评估分离比让模型自我批评更有效
4. **迭代假设检验**：每个 harness 组件都编码了某个假设，值得对新模型反复压力测试

作者总结：**harness 复杂度应该反映模型当下的真实局限。当局限随新版本变化时，有效的 harness 需要定期简化与重构。**
