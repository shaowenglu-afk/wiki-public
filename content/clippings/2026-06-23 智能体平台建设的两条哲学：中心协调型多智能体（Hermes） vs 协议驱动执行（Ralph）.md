# 智能体平台建设的两条哲学：中心协调型多智能体（Hermes） vs 协议驱动执行（Ralph）

**> 摘要：当一个团队真正开始建设长期、可治理的 Agent 平台时，首先要回答的不是"用什么框架"，而是"用什么样的协调哲学"。本文系统对比两条典型路线：一类是以 Hermes、LangChain 多 Agent 编排、AutoGen 为参考原型的"中心协调型多智能体"路线；另一类是受 Ralph pattern 启发、强调 fresh context、外部状态与强验收门禁的"协议驱动执行"路线。文章从 11 个工程维度做对比，提炼协议驱动执行的五条核心原则与七条工程化延展，并给出架构师在评审类似平台方案时可以反复追问的判断标准。

## 一、为什么这是一个必须先回答的问题

过去两年，Agent 的能力提升得非常快：代码补全、单元测试生成、需求澄清、影响面分析、文档撰写，几乎每一个工程环节都已经出现了"足够惊艳"的单点能力。但一个有意思的现象是：当一个团队真正把 Agent 落到长流程上时，会发现单点的"惊艳"并不能直接换来流水线的稳定。原因很简单——长流程由多个环节串成，每个环节都需要：明确的输入输出契约可审计的过程记录可回滚的失败恢复可沉淀的经验复用可治理的权限与审批边界这些特征都不是"模型能力"可以单独解决的，它们是流水线问题，是平台问题，更是工程哲学问题。在动手搭建任何 Agent 平台之前，必须先回答一个问题：你的 Agent 应该如何协作？**

市面上大致存在两种典型的工程哲学。它们代表了关于"Agent 应该如何协作"的两种不同答案。需要特别强调的是：这两条路线都可能包含多个 Agent，**关键的区别不在数量，而在协调方式、状态载体与推进机制**。

## 二、两条产品哲学

### 2.1 中心协调型多智能体路线

**核心隐喻**：一个会调度的"总指挥 Agent"，带着一组"专业 Agent"（澄清 Agent、检索 Agent、代码 Agent、写作 Agent、评审 Agent ……）协同工作，主要通过协调者把一切在长会话中串起来。

**参考原型**：Hermes、LangChain 多 Agent 编排、AutoGen 一类带有协调者/子 Agent 分工的框架或产品。

**核心特征**：

- **中心化协调**有一个**主 Agent（Orchestrator / Coordinator）** 负责拆分任务、调度子 Agent、汇总结果。整个系统的"大脑"是单一的。
- **角色化分工**子 Agent 通常扮演"研究员 / 工程师 / 评审员 / 写作者"等角色，靠 Prompt 定义人格与职责。
- **共享长会话**子 Agent 之间通过主 Agent 的会话上下文交换信息，而不是直接点对点通信。
- **弱边界**默认高权限、低审批，强调"少打断、少确认"。
**常见误解**：很多讨论会把注意力放在"有多少个 Agent"上，但真正的区别不在数量，而在协调中心是否外移、状态是否外部化，以及执行是否按最小单元推进：

> 这里说的不是"中心协调型路线只能有一个 Agent"，而是它更像"以协调者为中心的多 Agent 协作"。

**适用场景**：需求仍有较大不确定性、需要多个角色围绕同一上下文持续讨论和收敛方案的任务；也适合用于快速搭建多角色协作原型、研究型助手或交互型辅助系统。

### 2.2 协议驱动执行路线

**核心隐喻**：不是把一切压在一个会调度的总指挥身上，而是把任务拆成可以独立推进的最小执行单元，让系统围绕**执行计划、外部状态、验收门禁**持续循环。

**参考原型**：snarktank/ralph 及其背后的 Ralph pattern，一类以 fresh context、外部状态、单故事推进、质量门禁为核心的自治执行方式。

**核心特征**：

- **fresh context**每轮执行尽量使用新的上下文，避免长会话不断累积污染。
- **强协议**任务拆解、状态写回、验收推进尽量固化成显式规则，减少临场发挥。
- **强外部化**状态尽量放在会话之外，文件、Git、数据库都可以是真相源。
- **强门禁**是否推进下一步，不以"感觉做完了"为准，而以验证结果为准。
- **弱人格**不强调"AI 工程师"的人格形象，强调"系统能否稳定推进多步任务"。
**需要特别说明**：公开的 Ralph 更准确地说是"fresh-context 的自治循环"，而不是一个公开定义完备的"平权多 Agent 平台"。本文后面谈到的"群体协作"与"协议驱动多执行单元"，是吸收 Ralph 思想之后的工程化延展，而不是把 Ralph 原样照搬。

**若进一步做工程化延展，这一路线更适合的场景**：长期平台化生产、跨项目复用、多执行单元协作、强审计与治理、可持续演进的 Agent 平台。

### 2.3 一句话区分

> 中心协调型路线更像"协调者驱动的多智能体"，协议驱动执行更像"协议驱动的自治执行循环"。

这两种哲学本身并没有绝对的对错，但它们指向的工程目标明显不同：

- 中心协调型路线更关心"如何让协调者把多个角色组织起来"。
- 协议驱动执行更关心"如何让任务在外部状态与验收门禁下稳定推进"。
当目标是"长期、稳定、可治理的 Agent 平台"时，**协议驱动的工程优势会迅速放大**。原因很简单：协议可以被版本化、被审计、被继承，而协调者的"智能"往往会随着上下文变长而衰减。

## 四、为什么协议驱动执行更适合长期平台化

### 4.1 中心协调型路线的结构性脆弱

"中心化多智能体"路线最大的隐患，不在于"有没有多个 Agent"，而在于**协调智能本身的结构性脆弱**。一旦把任务拆分、调度、汇总、推进全部压到同一个协调者身上，这个协调者就会天然成为整条链路的单点中枢。可以归纳为五类典型风险：

- **协调者上下文污染**长会话跑过几十轮后，协调者开始忘记前面写过的约束，自己打自己的脸。
- **协调失败放大**跑到第 N 步崩溃时，**整条链路都不可恢复**——因为子 Agent 之间的中间结果都沉淀在协调者的会话里，没法独立取回。
- **自动执行失控**协调者拥有调度 + 工具调用 + 写文件的全部权限，一旦边界没设好，"顺手"做出的不可逆操作会影响所有下游子 Agent。
- **审计空白**想知道"协调者当时为什么调了这个子 Agent、为什么走到这一步"，只能拿到一段对话摘要，无法做决策级回溯。
- **流程漂移**协调者每一轮调度风格略有不同，下游测试和发布脚本无法稳定依赖。
这些未必是某个具体产品"参数没调好"的问题，而是这类路线在长链路场景下更容易暴露出的结构性风险：任务越长、上下文越长、权限越宽，协调者就越容易成为单点脆弱源。

> 中心协调型路线擅长做"协作智能"的体验，而流水线内核更需要"协议驱动的稳定"。两者并非绝对对立，但工程侧重点明显不同。

### 4.2 协议驱动的工程优势

把视线从"协调者能力"转移到"协议"上，工程优势会迅速显现：

- **协议可版本化**规则一旦结构化，可以放进 Git，可以打 tag，可以回滚。
- **协议可审计**每一次推进都有明确证据，不靠"我觉得做完了"。
- **协议可继承**同一套协议在不同项目、不同阶段可以叠加、合并、覆盖。
- **协议可并发**协议下的最小执行单元可以独立重试、独立失败、独立回滚。
- **协议可治理**权限、审批、成本统计都可以围绕"协议步骤"展开，而不是围绕"协调者会话"展开。

### 4.3 先稳内核，再做平台化

在 Agent 平台建设中，最常见的死法是"先做复杂治理，再做执行骨架"。结果是治理包住了一个不稳定内核，所有治理能力都建立在沙子上。

从协议驱动执行的思想里，可以提炼出一条对平台建设尤其重要的方法论：

> 先把最小执行闭环做稳，再补审批、观测、权限、成本统计。

这条路线让团队可以在任何阶段"先稳定再前进"，不会因一步错而全盘崩。它本质上是一句反共识：**治理能力不是前提，而是结果**。一个跑不稳的内核，包再多治理也只会更快暴雷。

## 五、协议驱动执行的五条核心原则

吸收不等于照搬。下面这几条，是协议驱动执行真正值得借鉴的核心思想。它们不是口号，而是可以落到具体产品形态和技术决策中的工程原则。

### 5.1 原则一：最小故事驱动

**核心做法**：反复强调"一次只做一个足够小的 story"，story 必须可独立判断通过与否。

**工程落地**：建立"最小执行单元"概念，每个单元是一类事情、一个输入、一个输出、一个验收标准。计划项状态采用六态机：pending / running / passed / failed / blocked / skipped。

**带来的收益**：

- 失败可定位：可以精确知道是哪一项失败、为什么失败
- 重试可执行：单步重试不会牵连已成功的步骤
- 回滚可边界：每一步都有清晰的"输入—输出—影响范围"

> 公开实现常用 prd.json 表达故事，工程实践中可以用结构化计划项表达故事。形式不同，本质一致。

### 5.2 原则二：外部化状态

**核心做法**：用外部状态文件 + Git 做真相源，绝不依赖长会话。

**工程落地**：状态尽量放在会话之外，数据库 + 文件 + Git 都可以是真相源。推荐"数据库为主状态源，文件为规则源与辅助记忆源"。状态变更必须在事务中完成"更新单元 + 写入审计 + 更新聚合"，任何一步失败全部回滚。

**带来的收益**：单步可恢复从"靠 Git 回滚"升级为"靠数据库事务回滚"，同时具备并发安全和分布式部署的扩展空间。

### 5.3 原则三：规划与执行分层

**核心做法**：强调"先有任务单，再执行"，规划与执行是两层不同的关注点。

**工程落地**：在服务层明确拆分两个独立模块：

- 计划生成模块：根据结构化需求生成执行计划，独立版本化
- 执行调度模块：根据计划项逐项调度执行
**带来的收益**：计划可以独立重建、独立升级，而执行不必跟着改。这让"调整执行策略"和"重构计划生成逻辑"可以并行演进。

### 5.4 原则四：规则文件化与策略合并

**核心做法**：把规则写到结构化文件（如 AGENTS.md、提示词模板）里，让规则可见、可改、可审计。

**工程落地**：更进一步，建立"**三级策略合并**"机制：

- 环境级默认策略
- 项目级策略
- 任务类型模板策略
由独立的合并模块负责合并，**合并过程本身可审计**。

**带来的收益**：同一个项目可以有多个阶段的策略；同一个阶段在不同项目里也可以有不同策略。这本质上是在把"Prompt 里塞规则"升级为"策略文件驱动规则"，从而显著降低规则漂移。

### 5.5 原则五：验收门禁

**核心做法**：每轮执行都要过质量检查；对前端故事，还会把浏览器验证显式写进 acceptance criteria。

**工程落地**：每个执行单元必带 acceptance_criteria，并由独立的质量模块执行结构检查、规则检查和证据校验。**未通过验收的单元只能进入 failed 或 blocked，不能直接跳到下一项**。

**带来的收益**：这是中心协调型路线最容易缺失、但生产环境最需要的能力。它让"自动化"升级为"有纪律的自动化"。

### 5.6 原则六：在循环执行思想上补足单步控制

**核心做法的局限**：协议驱动执行的公开实现更偏向循环执行。

**工程化补足**：进一步补上"单步执行"与"自动循环"两种控制模式，由人按阶段选择。

**典型落地**：在控制面提供四类命令：

- plan查看执行计划
- run-next单步执行下一项
- run-loop自动循环执行，直到完成或阻塞
- approve人工审批阻塞项
**带来的收益**：控制面不再只是一个"接口触发器"，而是升级为"编排观察器 + 编排控制器"，让领导、架构师、研发人员拥有同一个观测面与控制面。

## 六、协议驱动执行的七条工程化延展

如果只是照搬思想，工程原型没问题，但进入企业场景一定会撞墙。下面这些工程化延展，是把协议驱动执行从"个人/小团队工作流"推向"企业级 Agent 平台"必须补齐的能力。

### 6.1 延展一：数据库驱动，而非纯文件驱动

**纯文件驱动的局限**：以 JSON、纯文本文件 + Git 作为主状态载体，适合个人开发或小团队试验，但对多任务、多项目、多人协作的平台远远不够。

**工程化延展**：用数据库做主状态源，并且为每种状态变化提供**事务保障**。状态变更必须在事务中完成"更新单元 + 写入审计 + 更新聚合"，任何一步失败全部回滚。

**为什么必须延展**：没有事务保障，单步可恢复只是"看上去可恢复"。在真实多并发场景下，状态不一致会让协议驱动路线的稳定性优势归零。

### 6.2 延展二：独立的策略合并服务

**纯文件策略的局限**：规则写在文件里，但缺少一个"动态合并"的过程，规则只能覆盖、不能合并。

**工程化延展**：把策略合并做成一个**独立服务**，负责把环境级、项目级、阶段级策略合并成最终的执行规则，并输出**可审计的合并日志**。

**为什么必须延展**：在多项目、多团队、多阶段的真实企业场景下，策略必须支持"叠加、合并、覆盖、回退"四种语义。纯文件覆盖的做法在原型期没问题，但进入生产后一定会撞墙。

### 6.3 延展三：项目级记忆与任务级记忆分层

**单层记忆的局限**：仓库内 append-only 记忆适合承接单项目迭代中的经验，但并不天然适合作为跨项目、分层治理的长期记忆系统。

**工程化延展**：把记忆拆成**三种范围**：

- project项目级长期经验（如"此项目的某类生成任务前必须先调用某种澄清流程"）
- task任务级短期记忆（如"本次澄清中用户特别强调要兼容老用户"）
- global跨项目共享知识（如"前端接口联调文档必须包含字段变更说明"）
并通过独立的记忆服务提供"主动携带"语义——下游执行单元在执行前会主动查询上游相关记忆，而不是被动读取。

**为什么必须延展**：没有分层记忆，多项目复用经验时会出现严重的"经验串台"。一个 C 端项目的最佳实践如果污染到 B 端项目，会让整个平台的输出质量不可控。

### 6.4 延展四：计划模板化

**Prompt 化生成计划单的局限**：任务单生成更接近"Prompt + 少量示例"，生成质量不稳定。

**工程化延展**：建立计划模板目录，提供预制模板：

- 文档生成类任务
- 知识库同步类任务
- 新功能设计类任务
每种任务类型自动套用对应的计划模板，由计划生成模块调用对应模板生成执行单元。

**为什么必须延展**：在企业场景下，任务类型是有限的、可枚举的、可标准化的。把"执行单生成规则"从代码中抽离成模板，是协议驱动从"工程化原型"再推一步的关键。

### 6.5 延展五：分阶段实施路线

**没有演进路径的局限**：协议驱动思想本身没有告诉你"先做什么后做什么"，但对生产系统来说，实施顺序至关重要。

**工程化延展**：定义四级演进路线：

阶段

目标

关键交付
**协议骨架（Lite）**
建立执行计划骨架

最小执行单元、单步执行命令
**编排中心（Core）**
建立调度与策略合并

调度服务、策略合并服务
**记忆与质量（Pro）**
补齐分层记忆与质量门禁

分层记忆服务、质量门禁服务
**生产化增强**
平台治理

成本、审计、权限、模型路由

**为什么必须延展**：协议驱动思想给的是终态参考，不是演进路径。在企业场景下，**没有演进路径的架构方案等于没有方案**。团队必须根据自身代码基线、能力储备、业务节奏重新定义"先做什么、后做什么"。

### 6.6 延展六：审计事件流与可回放执行

**Git 审计的局限**：依赖 Git 历史做审计，但 Git 不擅长表达"为什么这次执行选择了这条路径"。

**工程化延展**：在审计事件表中记录**完整的决策与执行事件流**，包括：

- 计划项被选中的原因
- 命中的策略与合并结果
- 调用的技能及其入参出参
- 验收结果与失败原因
- 重试与降级决策
**为什么必须延展**：在企业 AI 平台中，"为什么 AI 做了这件事"比"AI 做了什么事"更重要。没有决策级审计，Agent 平台永远无法回答领导和合规部门的追问。

### 6.7 延展七：可治理的自动执行边界

**默认高权限自动执行的局限**：在企业场景下完全不可接受。

**工程化延展**：建立**四级自动执行边界**：

- **白名单 prompt 技能**默认自动执行（如纯澄清、纯规划类工具）
- **受控 prompt 技能**需通过策略校验后自动执行
- **脚本技能**默认进入人工审批，仅在特定阶段可自动执行
- **高风险操作**任何阶段都不允许自动执行，必须人工确认
**为什么必须延展**：Agent 平台最终要进入生产，而生产的第一原则是"少出事"，不是"多做事"。可治理的边界是平台能否真正进入生产的前提。

## 七、给架构师的五条判断标准

把上面所有分析收束起来，我认为至少有五条判断，值得在评审类似 Agent 平台方案时反复追问。

### 7.1 不要被"中心化多智能体"的性感叙事绑架

一个跑不稳的多智能体，往往比没有多智能体更糟。在评审 Agent 平台方案时，第一个问题不应该是"它有多少个 Agent"，而应该是：

> "这个方案在跑长链路任务时，第 N 步崩了，系统能不能精确回答是哪一步崩的、为什么崩的、可不可以只重试这一步？"

如果答案是"不能"，那无论方案叙事多么性感，都应该保持警惕。

### 7.2 状态必须在会话之外，否则你永远无法解释"昨天到底发生了什么"

Agent 平台的真相源，必须落在 **数据库 + 文件** 这一类会话之外的载体上，而不能寄希望于会话历史。原因很直接：

- **可恢复性**会话是不可恢复的，数据库可以。
- **可审计性**会话无法支持"为什么这样做"的反查。
- **可并发性**会话无法支持多人同时推进。
- **可治理性**会话无法支持权限与审批。

### 7.3 规则必须文件化、版本化、可合并

只靠 Prompt 维护规则，规则迟早会漂移。要把规则真正变成工程资产，至少要落到三件事：

- **结构化文件**JSON / YAML，而不是 Prompt 散文。
- **版本化**每一次策略变更都可追溯。
- **可合并**支持环境级、项目级、阶段级策略合并，而不是简单的覆盖。

### 7.4 验收必须成为必经路径

"做完"不等于"完成"。每一个执行单元都必须定义 acceptance_criteria，并且由独立服务执行验收；否则自动化不会收敛错误，只会更快地放大错误。

### 7.5 先稳内核，再做平台化增强

没有最小闭环的治理，本质上只是在包住一个不稳定内核。更可靠的演进顺序永远是：

> 执行计划骨架 → 编排中心 → 记忆与质量 → 平台治理。

反过来做，治理越多，内核崩得越快。

## 八、写在最后：不是造一个总指挥，是建一支协议驱动的群体

刚接触 Agent 平台时，很多人都曾幻想过：是不是只要造出一个足够聪明、足够会调度的总指挥，就能把一切都串起来。

但今天，这个答案已经越来越清晰：

> Agent 平台的未来，不是继续放大单个中心化大脑，而是建设一支能够被协议稳定驱动的执行群体。

不是继续培养一个既能写代码、又能改 Bug、还能做测试、顺手上线部署的全能"总指挥 Agent"，而是搭建一支由**需求澄清器、代码分析器、方案生成器、质量检查器、测试用例生成器、影响面分析器**组成的执行群体。

这些 Agent 单独看都未必足够"聪明"，但只要它们在清晰的协议下协作，系统就能跑得很稳，也能跑得很远。

这条路，**受 Ralph pattern 启发，但绝不止于 Ralph pattern**。

中心协调型路线让人更清楚地看到"协调智能"在长链路上的边界，协议驱动执行则给了一套"协议驱动稳定推进"的方法论。而 **真正的 Agent 平台建设，最终还是要把方法论翻译成企业级工程能力**。这一步，恰恰是最值得原创、也最值得投入的部分。
