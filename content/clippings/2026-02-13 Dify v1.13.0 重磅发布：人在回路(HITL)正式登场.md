# Dify v1.13.0 重磅发布：人在回路(HITL)正式登场

![](https://mmbiz.qpic.cn/mmbiz_png/yeBwK8vYkibNv1W2TLuQOtjUbsZRzjDjXehPPUrrz8fQX34jJLKtGmPsVScW5Fo4zes8ibHMAdlNCZicloUpxWQIQpHgWeUwZEwm3QkofRxqxA/640?wx_fmt=png&from=appmsg)

# 

**"AI 不是替代人，而是让人成为决策链上的关键节点。"**

Dify 社区苦等已久的 **Human-in-the-Loop（人在回路）** 功能终于在 v1.13.0 正式版中亮相！这不仅是功能更新，更是 Dify 从"自动化工具"向"人机协作平台"转型的里程碑。本文从实战角度，为你拆解这个版本的硬核更新与升级注意事项。

## 一、革命性新功能：Human-in-the-Loop（人在回路）

### 1. 为什么需要 HITL？

在实际业务场景中，完全自动化的 AI 工作流往往存在"信任鸿沟"：金融审批、医疗诊断、内容发布等高风控场景，既需要 AI 的处理速度，又离不开人类的关键决策。v1.13.0 新增的 **Human Input 节点** 完美解决了这个痛点。

### 2. 实战应用场景

- **内容审核流**：AI 生成营销文案 → HITL 节点（人工审核/修改）→ 正式发布
- **智能客服升级**：AI 处理客户投诉 → 敏感问题自动转人工确认 → 继续后续流程
- **数据标注验证**：模型预测结果 → 人工抽样复核 → 合格数据入库

### 3. 核心能力详解

- **原生工作流暂停**：在关键决策点插入 Human Input 节点，工作流会自动挂起等待输入
- **可视化审核界面**：系统自动生成 UI 界面，审核人可直接查看 AI 输出并修改变量（如编辑草稿、修正数据）
- **动作路由**：支持自定义按钮逻辑（"批准"、"拒绝"、"升级"），不同动作触发不同分支
- **灵活交付**：支持通过 **Webapp 界面** 或 **邮件** 发送审核表单（云版本邮件功能视套餐而定）
- **超时机制**：可配置全局超时时间（默认 7 天），超时后自动清理挂起任务
**环境变量配置提醒**：

HUMAN_INPUT_GLOBAL_TIMEOUT_SECONDS=604800  # 默认7天**ENABLE_HUMAN_INPUT_TIMEOUT_TASK=true# 启用超时检查HUMAN_INPUT_TIMEOUT_TASK_INTERVAL=1        # 检查间隔（分钟）

## 二、工作流执行引擎架构升级：为生产环境而生

v1.13.0 对工作流执行架构进行了深度重构，这对大规模自托管部署至关重要。

### 1. Celery 化改造

流式执行迁移**：基于工作流的流式执行（Streaming）和高级对话执行（Advanced Chat）现已移至 Celery Worker 中运行

- **非流式保持**：非流式的 WORKFLOW 执行仍在 API 进程中运行
- **事件驱动**：所有暂停/恢复路径（包括 HITL）均通过 Celery 恢复，事件通过 Redis Pub/Sub 流回前端

### 2. 新增必需队列

**⚠️ 升级必看**：v1.13.0 引入了新的 Celery 队列 workflow_based_app_execution，**必须在部署配置中添加**，否则流式执行和恢复任务将无法处理！

如果你使用自定义 CELERY_QUEUES，请确保包含：

"workflow_based_app_execution",  # 必需：处理基于工作流的执行**"api_token"# 可选：如启用 API Token 使用时间更新任务

### 3. Redis Pub/Sub 集群支持（高并发必配）

针对高吞吐量场景，v1.13.0 引入了专业的 Pub/Sub 配置：# 关键环境变量PUBSUB_REDIS_URL=redis://your-redis-cluster:6379/1  # 独立 Redis 实例建议PUBSUB_REDIS_CHANNEL_TYPE=sharded                   # 高吞吐环境强烈推荐PUBSUB_REDIS_USE_CLUSTERS=true# 启用 Redis 集群模式实战建议**：

- 小规模部署：使用默认配置即可，Dify 会自动复用现有 Redis
- **大规模/生产级部署**：务必配置独立的 PUBSUB_REDIS_URL，并启用 Sharded PubSub + Redis Cluster，这是水平扩展的关键！

## 三、性能与可靠性：看不见的硬核优化

### 1. 数据库性能提升

- **异步写入**：数据库写入操作改为异步非阻塞，复杂工作流（特别是并行分支）执行时间大幅缩短，总耗时现在接近最长分支而非各分支之和
- **流式输出优化**：超时问题显著减少，输出更稳定
- **索引优化**：消息清理性能提升，批量删除采用分片策略避免 DB 压力峰值

### 2. 关键 Bug 修复

- **租户默认模型重复**：新增唯一约束，彻底解决迁移时的重复记录问题
- **工具删除异常**：修复 Provider ID 类型不匹配导致的工具删除失败
- **FastOpenAPI 认证回退**：修复远程文件 API 中已认证用户被解析为匿名用户的问题
- **ECharts 安全加固**：移除动态 new Function 执行，防止潜在 XSS 风险

### 3. MCP 与 API 增强

- **MCP 工具使用追踪**：现在可从 MCP 响应中提取 Token/成本等元数据
- **Service API 新增**：支持终端用户查找端点，并强制租户/应用范围验证
- **工作流历史刷新**：运行状态转换时的历史记录刷新行为优化

## 四、无痛升级指南

### Docker Compose 部署（推荐）

# 1. 备份（可选但强烈建议）**cd dockercp docker-compose.yaml docker-compose.yaml.$(date +%s).baktar -cvf volumes-$(date +%s).tgz volumes# 2. 拉取最新代码git checkout maingit pull origin main# 3. 停止服务docker compose down# 4. 启动（注意：如有 PostgreSQL 启动问题，使用以下命令）docker compose --profile postgresql up -d⚠️ 关键检查点**：

- 确保 docker-compose.yaml 中 Worker 服务已订阅 workflow_based_app_execution 队列
- 检查是否配置了 PUBSUB_REDIS_URL（高并发场景）

### 源码部署

# 1. 停止服务**# 2. 切换分支git checkout 1.13.0# 3. 更新依赖（注意：v1.13.0 已全面迁移至 uv）cd apiuv sync# 4. 执行数据库迁移uv run flask db upgrade# 5. 重启服务依赖管理变更**：v1.13.0 已完成从 Poetry 到 uv 的全面迁移，升级时请确保使用 uv 而非 poetry！

## 五、实战建议：什么时候必须升级？

- **需要人工审核环节**：如果你正在用 Dify 构建需要"AI 初筛+人工确认"的业务流，v1.13.0 是必升版本，无需再外挂复杂的回调系统
- **高并发场景**：流式执行移至 Celery 后，API 进程负载大幅降低，配合 Sharded PubSub 可支撑更大规模并发
- **稳定性要求高**：数据库异步写入和连接池优化，显著减少复杂工作流的超时和连接错误
**暂缓升级的情况**：

- 如你使用重度自定义 Celery 配置，需先评估新队列的集成成本
- 如对 Redis Pub/Sub 有严格网络隔离要求，需先规划独立 Redis 实例

## 结语

Dify v1.13.0 不仅是一个功能版本，更是架构层面的成人礼。Human-in-the-Loop 的引入让 AI 工作流真正具备"人机协作"能力，而 Celery 化和 Redis Pub/Sub 的集群支持，则标志着 Dify 已经做好了服务大型生产环境的准备。
