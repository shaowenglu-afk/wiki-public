# 智能体的记忆，不是把聊天记录塞进提示词

> 这是一篇面向工程落地的智能体记忆系统整理。把短期上下文、长期记忆、检索注入、项目上下文、操作轨迹和治理机制放在一起，整理成一套更接近生产实践的工程模型。

很多智能体演示看起来很聪明，但一到生产环境就像“金鱼记忆”。

用户第一轮说：“我在杭州，周末出门主要看降雨概率，不太关心风力。”

十几轮之后，用户问周末要不要带伞，智能体却查了北京天气，只复述了最高温，完全没有提降雨概率。

这时模型没坏，工具也没坏。真正的问题是：系统从来没有设计过“什么应该记住、应该记多久、下次怎么找回来”。

所以，智能体的记忆不是一个简单的功能开关，也不是把完整聊天记录一直塞进提示词。它更像一个生产系统里的状态管理子系统：

Short-Term Memory 负责当前对话不断片。
Long-Term Memory 负责跨会话保留稳定事实。
Retrieval 负责在合适的时候把相关记忆找回来。
Governance 负责让这些记忆可见、可删、可审计。

如果这些边界没有想清楚，所谓“记忆”很容易变成一个又贵、又不准、还带隐私风险的缓存。

## 为什么智能体需要专门的记忆系统

普通聊天机器人通常只要回答当前问题。只要上下文还在窗口里，它就能表现得不错。

但智能体不一样。它通常要处理更长的任务、更复杂的工作流，以及跨会话的用户偏好。

比如：

- 用户长期偏好中文回答
- 用户看天气时更关心降雨概率
- 用户上次提交过一个退款工单
- 用户某个项目里已经做过某个决策
- 某个工具调用顺序在这个业务里更可靠
这些信息不一定都适合塞在当前提示词里。有些只在当前对话有效，有些应该跨天保留，有些还涉及隐私和删除权。

所以智能体记忆的核心问题不是“存不存聊天记录”，而是：

什么信息应该成为状态？
这个状态属于当前会话，还是属于长期用户画像？
这个状态什么时候召回？
召回后怎么放进模型上下文？
用户能不能查看和删除？

这才是从演示走向生产时真正要解决的问题。

## 记忆不是一种东西

很多人提到记忆，会下意识把它当成一个整体。但在智能体系统里，记忆至少要拆成四类：

Short-Term Memory（短期记忆）：当前对话正在用的信息
Episodic Memory（情景记忆）：过去发生过什么
Semantic Memory（语义记忆）：稳定事实和偏好
Procedural Memory（程序性记忆）：学到的流程和做法

这几类记忆的生命周期、存储方式和召回机制都不一样。

### Short-Term Memory（短期记忆）：当前对话不断片

Short-Term Memory，也就是 STM，通常存在于模型的上下文窗口里。

它适合保存：

- 最近几轮对话
- 当前任务状态
- 本轮会话里的临时条件
- 用户刚刚提供的信息
比如用户说：

这次旅行预算控制在 5000 元以内。

如果这个条件只对当前规划任务有效，它就是 Short-Term Memory。

STM 的特点是写入简单，但保留成本高。因为只要放进提示词，每一轮都要付 token 成本。对话越长，成本越高，窗口也越容易被挤满。

### Episodic Memory（情景记忆）：记录发生过什么

Episodic Memory 记录的是事件。

比如：

用户上周提交过退款工单 #4412。
用户昨天确认了方案 B。
用户之前让系统把某个客户标记为高优先级。

这些不是稳定偏好，而是过去发生过的事情。

它们通常不应该一直塞在上下文里，而应该存到模型外部，需要时再召回。

### Semantic Memory（语义记忆）：稳定事实和偏好

Semantic Memory 保存更稳定的信息。

比如：

- 用户姓名
- 用户所在城市
- 用户偏好中文回答
- 用户看天气时更关心降雨概率
- 公司固定税率
- 某个产品的长期配置
这类记忆通常小而常用，经常会被注入到系统提示词或专门的“用户信息块”里。

### Procedural Memory（程序性记忆）：学会怎么做

Procedural Memory 记录的是流程和做法。

比如：

当用户查询退款状态时，先查订单，再查退款系统，最后合并解释。

这类记忆更像“经验”或“技能”。它可能表现为更新后的系统提示词、少样本示例、技能库，或者某种固定工具调用模式。

所以智能体记忆不是一个大篮子。不同类型的记忆，应该进入不同的数据结构和不同的生命周期。

## STM 和 LTM 怎么分

一个实用的判断标准是：

如果一个事实只在当前对话里有效，它属于短期记忆。
如果下次用户打开应用时它仍然有效，它属于长期记忆。

例如：

- “用户刚才选择了方案 B”通常是短期记忆
- “用户习惯用中文回答”通常是长期记忆
- “用户这次上传了一个 PDF”通常是短期记忆
- “用户所在公司默认税率是 13%”可能是长期记忆
这个判断不是技术细节，而是产品设计问题。

因为一旦你决定它是长期记忆，就意味着系统要考虑：

- 存在哪里
- 保留多久
- 是否需要用户授权
- 是否允许用户查看
- 是否允许用户删除
- 出错后如何追踪
长期记忆不是免费能力，它是一份状态管理责任。

## 为什么不能只靠完整聊天记录

最简单的记忆方案是：每一轮都把完整聊天记录重新放进提示词。

这个方案在演示里很有诱惑力，因为它几乎不用设计。对话很短时，模型确实能看见所有历史。

但生产环境里它很快会失败。

### 上下文窗口会被挤满

智能体的提示词里不只有聊天记录，还可能有：

- 系统提示词
- 工具定义
- RAG 检索文档
- 工具返回结果
- 多智能体协作过程
- 历史对话
哪怕模型支持很长上下文，也很容易被这些内容占满。

一旦窗口滚动，早期信息就会被丢掉。

比如用户第 2 轮说了航班号，第 16 轮再问航班落地时间。如果系统只保留最近几轮上下文，航班号可能已经不在窗口里。

### Token 成本会持续膨胀

完整聊天记录不是一次性成本，而是每轮重复支付。

历史越长，每一轮越贵。

这就是为什么“全量缓冲”不适合作为生产默认策略。它适合短流程、调试、评测，但不适合长期在线服务。

### 长上下文不等于好召回

即使某个事实还在上下文窗口里，模型也不一定能稳定用到。

Lost in the Middle 这类研究指出，模型在长上下文中往往更关注开头和结尾，中间信息容易被弱化。

所以记忆系统不能只追求“塞进去”，还要考虑“能不能被准确召回和使用”。

## 短期记忆的四种策略

生产系统里，短期记忆通常有四种做法。

### 1. 全量保留

也就是保留所有对话原文。

优点很明显：

- 信息不丢
- 实现简单
- 适合调试和评测
缺点也很明显：

- token 成本不断增长
- 上下文容易爆
- 长上下文召回质量可能下降
所以它只适合短流程，不适合默认上线。

### 2. 滑动窗口

只保留最近 N 轮对话，旧内容直接丢掉。

优点是成本可控，实现简单。

缺点是早期关键事实可能消失。

例如用户一开始说了航班号，后面再问航班状态，系统可能已经看不到那个航班号了。

### 3. 滚动摘要

把较早的对话压缩成摘要。

优点是能保留早期信息，同时减少token消耗。

缺点是摘要有损，摘要质量会影响后续回答。而且它需要额外模型调用。

但这个模型调用不一定要用最强模型。很多时候，用便宜的小模型做摘要就足够了。

### 4. 混合策略：最近原文 + 旧内容摘要

更推荐的生产默认方案是：

最近 N 轮对话保留原文。
更早内容压缩进滚动摘要。

这样有几个好处：

- 最近上下文保持精确
- 早期关键信息不会完全丢掉
- token 成本可控
- 对话变长时退化更平滑
如果不知道该选什么，优先从这个策略开始。

## 长期记忆是一条数据管线

长期记忆不是“把所有聊天记录存起来以后搜索”。

更合理的结构是一条管线：

抽取 -> 存储 -> 召回 -> 注入 -> 反馈

### 第一步：抽取

每轮对话结束后，系统用一个小模型或结构化输出提示词，扫描这轮交互里有没有值得长期保存的信息。

适合抽取：

- 稳定事实
- 用户偏好
- 用户决策
- 长期指令
应该跳过：

- 寒暄
- 重复确认
- 临时上下文
- 原始隐私信息
- 未确认事实
长期记忆抽取应该保守，而不是贪多。

如果系统什么都记，最后记忆库会变成噪声库。

### 第二步：存储

一条生产级长期记忆不应该只有一段文本。

它至少应该包含：

用户 ID
智能体 ID
记忆类型
记忆内容
向量表示
关键词
重要性评分
使用次数
最近使用时间
创建时间

这些字段不是形式主义。

用户 ID 用来防止跨用户泄露。智能体 ID 用来防止不同智能体误共享记忆。类型字段用来区分事实、偏好、事件、指令。向量和关键词用于召回。使用次数和最近使用时间用于排序、衰减和清理。

如果没有这些元数据，后面就很难做可靠召回、审计和删除。

### 第三步：召回

用户下一次请求时，系统需要从长期记忆里找出相关内容。

生产系统里不应该只依赖纯向量召回，更稳的做法是混合召回：

关键词匹配 + 向量相似度 + 最近性 + 使用次数

原因很简单：向量搜索擅长语义相似，但对订单号、SKU、用户 ID、专有名词不一定稳定。

关键词匹配能补足精确匹配。最近性可以提高近期事实权重。使用次数能让真正有用的记忆上浮。

### 第四步：注入

召回后，把少量相关记忆注入到模型上下文。

例如：

你记得的用户信息：
- 用户常住城市是杭州。
- 用户看天气时更关心降雨概率。

这里要克制。

不要把几十条记忆都塞进去。不要注入无关记忆。不要注入过期记忆。更不能注入其他用户的记忆。

长期记忆不是越多越好。注入到提示词里的，应该是少量、相关、可信的内容。

### 第五步：反馈

如果某条记忆被召回并注入，就更新它的使用信息：

使用次数 + 1
最近使用时间 = 当前时间

这样常用记忆会上浮，长期不用的记忆会下沉，最后被清理。

这一步很重要。没有反馈，记忆系统就不知道哪些内容真的有用。

## 记忆必须按用户和智能体隔离

这里有一个关键判断：

记忆应该按“用户 + 智能体”限定范围。

也就是说，长期记忆至少要按两个维度隔离：

- 用户
- 智能体
不要让所有用户共享一个长期记忆库。

这不是代码组织问题，而是隐私边界。

跨智能体共享记忆也要谨慎。只有当多个智能体属于同一个产品界面、同一个用户授权上下文时，才考虑共享。否则应该按智能体 ID 分命名空间。

## 记忆要有上限和衰减

长期记忆不能无限增长。

一个实用做法是：给每个 (用户, 智能体) 设置硬上限。

例如：

每个用户在每个智能体下最多保留 200 条长期记忆。

清理时可以综合考虑：

重要性评分 × 最近性 × 使用次数

低分、过期、从未被召回的记忆，应该逐渐删除。

否则记忆库会越来越吵，召回质量反而下降。

## 记忆不是魔法，是用户状态

我认为最重要的判断是：记忆应该被当成用户状态，而不是框架魔法。

既然是用户状态，就应该：

- 可查看
- 可删除
- 可审计
- 可回放
- 可评测
用户有权知道系统记住了什么，也应该能要求删除。

这不是一个以后再补的功能，而是从第一天就应该设计的能力。

尤其在真实产品里，记忆系统可能会保存偏好、个人信息、历史决策、业务状态。只要系统记住了，就要能解释、能删除、能追踪来源。

## 记忆系统也需要测试

记忆系统不能靠感觉。

可以维护一组固定召回测试集，也就是专门验证记忆召回能力的测试集。

例如：

问题：用户看天气时更关心什么？
期望：降雨概率

问题：用户上次退款订单号是什么？
期望：#4412

问题：用户常住哪个城市？
期望：杭州

每次改提示词、改摘要策略、改召回逻辑、改模型版本，都跑一遍。

可以观察：

- Recall@1
- Recall@K
- 回答是否正确
- 注入的记忆是否正确
如果记忆召回能力下降，应该阻塞发布。

这和代码测试是一样的。记忆系统如果悄悄退化，用户只会感觉“这个智能体怎么越来越不懂我”。

## 怎么判断是记忆问题还是模型问题

调试记忆系统时，有一个很实用的判断：

如果事实出现在链路追踪里，说明模型拿到了它。
如果事实没有出现在链路追踪里，说明记忆管线在模型之前就失败了。

也就是说，先看模型实际看到的提示词。

如果相关记忆没有被注入，那问题可能在抽取、存储、召回或排序。

如果相关记忆已经被注入，但模型没用好，那才是模型推理或提示词设计问题。

这个区分很重要，否则你会把召回问题误判成模型能力问题。

## 真实智能体通常会把记忆拆成五层

如果只从抽象概念出发，很容易把记忆理解成“短期 + 长期”两层。但看真实的智能体和代码智能体实现，会发现生产系统通常会把记忆拆得更细。

我更倾向于用下面这个五层模型理解：

会话上下文：当前轮对话、工具结果、任务状态
压缩上下文：滚动摘要、上下文压缩、阶段性任务摘要
项目上下文：仓库结构、项目约定、配置文件、文档
长期用户记忆：用户偏好、稳定事实、历史决策
操作轨迹：工具调用、文件修改、错误、回滚、评测结果

这五层不应该混在一个向量库里。

会话上下文追求的是“当前任务不断片”。压缩上下文追求的是“窗口快满时不要丢掉关键事实”。项目上下文追求的是“智能体知道自己在哪个工程里工作”。长期用户记忆追求的是“跨会话保持连续性”。操作轨迹追求的是“可调试、可回放、可审计”。

如果系统没有这层拆分，后面很容易出现几类问题：

- 把临时事实误写入长期记忆
- 把项目规则当成用户偏好
- 把工具错误结果当成可信事实
- 把历史轨迹直接塞进提示词，导致上下文变脏
- 召回时只看语义相似，忽略文件路径、时间、权限和任务阶段
生产级记忆系统的第一步，不是选向量数据库，而是先定义这几类状态的边界。

## 从开源智能体里能学到什么

不同项目对“记忆”的命名不一样。有些直接叫 memory，有些叫 context、state、checkpoint、trajectory、repo map。名字不同，但背后的工程问题很接近：有限上下文里，系统要决定哪些信息应该保留、压缩、检索和注入。

### Claude Code：把项目记忆显式放进文件

Claude Code 的一个重要设计是把“记忆”分成两类文件。

第一类是 CLAUDE.md。它更像项目级操作手册：团队可以把构建命令、测试方式、代码风格、仓库约定、常见坑点写进去，让智能体在每次进入项目时自动获得这些上下文。

第二类是 auto memory 使用的 MEMORY.md 和相关主题文件。MEMORY.md 更像一个紧凑索引，启动时加载一部分；更详细的调试经验、架构笔记、接口约定可以被拆到其他文件里，智能体需要时再读。

这个设计给我们的启发是：

不要把所有长期知识都藏在不可见的向量库里。
稳定、重要、团队共享的知识，应该能被人直接读、直接改、直接做版本管理。
自动积累的经验，也应该能被人审计、修剪和删除。

对企业内部智能体来说，这一点很关键。比如“退款必须先查订单状态”“生产数据库只读”“提交前必须跑某个测试命令”，这些与其作为模糊记忆被召回，不如作为显式项目规则或组织规则进入版本库。

这类记忆更接近 Procedural Memory，但它的存储形态不是向量，而是可审查的文本文件。

### OpenClaw：用文件和检索把记忆分层

OpenClaw 的记忆设计很适合拿来说明“可见、可编辑、可检索”的价值。它的文档强调：模型只会记住被保存到磁盘上的内容，不存在隐藏状态。长期事实、偏好和决策进入 MEMORY.md；当天运行过程中的观察和会话摘要进入 memory/YYYY-MM-DD.md 这类日记文件；更大的历史内容通过检索工具按需找回。

这个设计的关键不是“把聊天记录都保存起来”，而是把记忆拆成不同热度：

- MEMORY.md：紧凑、长期、启动时注入
- 日记文件：更详细的工作层，适合记录观察和阶段性摘要
- 检索索引：需要时通过关键词、BM25 或向量召回
这对生产系统很有启发：长期记忆不应该只有一个数据库表，也不应该只有一个向量库。高频稳定信息可以直接进入启动上下文，低频详细信息应该留在外部，通过搜索召回。

对复杂工具型智能体来说，STM 也不能只是最近几轮对话，还应该包括结构化任务状态。比如：

目标：修复登录失败问题
已确认：前端请求返回 401
已排除：不是网络错误
待确认：后端令牌校验逻辑
最近失败操作：测试环境缺少环境变量

这种结构化状态比一大段对话摘要更可靠，也更容易被测试。

### Hermes Agent：把内建记忆和外部记忆提供方分开

Hermes Agent 的公开文档里有一个很实用的边界：内建记忆由 MEMORY.md 和 USER.md 组成，分别保存智能体学到的环境、约定和用户画像；外部记忆提供方则作为插件接入，可以负责跨会话检索、用户建模、图谱、向量搜索或混合搜索。

这说明生产系统里可以把记忆拆成两层：

内建记忆：小、稳定、始终可见。
外部记忆：大、可检索、按需注入。

Hermes 的外部记忆提供方还会把一部分能力暴露成工具，例如搜索、写入、更新、删除和反馈。这样一来，记忆就不再是不可观察的后台魔法，而是可以像其他工具调用一样被记录、限权、审批和评测。

但这也带来风险：如果完全让模型决定何时写入，可能会把噪声写进长期记忆。所以更稳的做法通常是：

模型提出候选记忆。
系统用规则和结构化校验过滤。
必要时让用户确认。
通过后再写入长期存储。

这比“每轮对话后自动抽取并写库”更慢，但对生产环境更安全。

### LangGraph：把线程状态和长期存储分开

LangGraph 的设计很值得参考，因为它把两件事分开了：

Checkpointer：保存线程级会话状态
Store：保存跨线程长期记忆

这个拆分非常接近生产系统的真实需要。

当前线程里的状态可以频繁变化，可以回放，可以恢复。跨线程长期记忆则应该更稳定，更严格地按用户、命名空间和权限隔离。

很多系统的问题就在于把这两者混在一起：把当前任务的临时状态写进长期记忆，或者用长期记忆承担会话恢复。前者会污染用户画像，后者会导致恢复不完整。

### Letta：把内存块和归档记忆分开

Letta 的记忆模型也很有参考价值。它把智能体的上下文分成更靠近提示词的内存块，以及更外部、更大规模的归档记忆。

这提示我们：长期记忆也不一定只有一种热度。

可以分成：

热记忆：少量、高频、直接进入上下文
温记忆：需要召回后进入上下文
冷记忆：只用于审计、回放或离线分析

用户偏好、身份相关信息可能是热记忆。历史工单和旧项目决策可能是温记忆。完整操作轨迹和旧对话原文更适合冷存储。

### Aider：代码智能体需要代码结构记忆

Aider 的 repo map 思路说明了一件事：对代码智能体来说，最重要的记忆未必是用户偏好，而是代码结构。

当一个仓库很大时，智能体不可能把所有文件都塞进上下文。它需要一个压缩过的项目地图，知道：

- 哪些文件重要
- 哪些符号在哪里定义
- 当前修改可能影响哪些模块
- 哪些上下文应该优先进入提示词
这类记忆不属于传统用户长期记忆，更像项目级 Semantic Memory。它通常来自静态分析、文件索引、代码搜索和最近编辑记录，而不是来自聊天记录。

这对所有企业级智能体都有启发：不同领域需要不同的结构化记忆。代码智能体需要 repo map，客服智能体需要客户画像和工单轨迹，数据分析智能体需要数据字典和指标口径。

### OpenHands 和 SWE-agent：操作轨迹本身就是记忆

OpenHands、SWE-agent 这类软件工程智能体都非常重视行动轨迹。它们面对的是一个持续变化的环境：文件会被修改，命令会执行，测试会失败，错误会被修复。

在这种场景里，记忆不仅是“用户说过什么”，还包括：

- 执行过什么命令
- 修改过哪些文件
- 哪个测试失败了
- 错误信息是什么
- 哪个修复尝试无效
- 最后通过了什么验证
这些轨迹不一定都要进入提示词，但必须能被回放和检查。

对生产系统来说，这类轨迹是可观测性的一部分。没有它，你很难回答“智能体为什么这么做”“它基于什么上下文做了这个决定”“问题发生在哪一步”。

## 生产级记忆系统应该怎么落地

综合这些系统，我认为生产级智能体记忆可以按下面的结构落地：

1. 显式上下文：系统提示词、项目规则、用户可编辑偏好
2. 会话状态：当前任务、最近对话、工具结果、未完成步骤
3. 压缩摘要：滚动摘要、阶段性总结、失败尝试摘要
4. 可检索记忆：长期事实、偏好、历史事件、项目知识
5. 审计轨迹：工具调用、召回记录、注入记录、输出和反馈

其中，最容易被低估的是“显式上下文”和“审计轨迹”。

显式上下文决定系统稳定性。凡是高优先级、长期有效、团队共享的规则，都不应该只依赖向量召回。它应该有明确文件、配置或数据库字段。

审计轨迹决定可运维性。每次记忆写入、召回、注入和删除，都应该能查到来源。否则当智能体输出错误答案时，你不知道问题来自哪里。

一个比较稳的写入流程可以是：

候选抽取 -> 类型分类 -> 敏感信息过滤 -> 冲突检测 -> 作用域判断 -> 写入或等待确认

召回流程可以是：

任务分析 -> 作用域过滤 -> 关键词召回 -> 向量召回 -> 时间和使用频率排序 -> 权限过滤 -> 注入

注意这里有两个过滤必须放在前面：

- 作用域过滤：先确定用户、智能体、项目和租户边界
- 权限过滤：只召回当前动作有权使用的信息
不要先从全库召回，再在结果里过滤。那样不仅浪费，也更容易造成越权风险。

## 几个框架里的记忆设计

不同智能体框架对记忆的抽象不太一样。

### LangChain

LangChain 的记忆组件比较显式，也比较可组合。

典型组件包括：

- ConversationBufferMemory
- ConversationSummaryMemory
- ConversationBufferWindowMemory
- VectorStoreRetrieverMemory
它适合用来理解记忆系统的基本构件。你可以组合短期摘要和向量存储，但也需要自己处理边界、作用域和注入方式。

### LangGraph

LangGraph 的设计更接近生产架构，这一点前面已经提到过。

它把记忆拆成两部分：

Checkpointer：保存线程级会话状态
Store：保存跨线程长期记忆

这种拆分比较清晰：当前会话状态和跨会话长期事实不混在一起。

如果要做长期运行的智能体工作流，这种模型更稳。

### CrewAI

CrewAI 更偏开箱即用。

设置：

memory=True

就会启用短期记忆、长期记忆和实体记忆。

它适合快速做多智能体工作流，但如果你要做强隔离、可审计、可删除的生产记忆系统，仍然需要检查它的默认行为是否满足你的要求。

### OpenAI Agents SDK

OpenAI Agents SDK 的会话机制更偏短期记忆和会话连续性。

同一个会话 ID 可以复用会话状态。

但长期记忆还需要你自己接：

- 向量存储
- 记忆抽取
- 召回
- 注入
- 删除和审计
所以会话机制不能替代完整的长期记忆管线。

### Strands

Strands 的模式是：

对话管理器负责短期记忆
记忆工具负责长期记忆

智能体可以通过工具调用主动存取记忆。

这个设计的关键点是：记忆本身也变成了一种工具能力，需要和其他工具一样考虑权限、边界和可观测性。

## 一份生产自查清单

如果你正在做智能体记忆系统，可以用下面这份清单自查。

### 显式上下文

- 项目规则是否放在可审查、可版本管理的位置？
- 高优先级规则是否避免只依赖向量召回？
- 用户偏好是否有可编辑入口？
- 团队共享约定和个人偏好是否分开？
- 系统提示词、项目规则和长期记忆的优先级是否明确？

### 短期记忆

- 是否明确选择了短期记忆策略？
- 是否避免把全量聊天记录作为默认方案？
- 是否使用“最近原文 + 滚动摘要”的混合策略？
- 摘要模型是否可以用更便宜的小模型？
- 摘要质量是否有评估？
- 当前任务状态是否结构化保存，而不是只靠自然语言摘要？

### 长期记忆抽取

- 是否只抽取长期有效的信息？
- 是否跳过寒暄、重复确认和临时事实？
- 是否拒绝原始隐私信息？
- 是否使用结构化输出？
- 是否有重要性评分？
- 是否做冲突检测，比如新偏好和旧偏好冲突？
- 是否区分模型候选记忆和已经确认的记忆？

### 长期记忆存储

- 每条记忆是否包含用户 ID？
- 每条记忆是否包含智能体 ID？
- 每条记忆是否包含项目、租户或组织作用域？
- 是否区分事实、偏好、事件、指令？
- 是否保存向量表示？
- 是否保存关键词？
- 是否保存使用次数？
- 是否保存创建时间和最近使用时间？
- 是否保存来源，比如来自哪次对话、哪个工具结果或哪个文件？

### 召回和注入

- 是否使用关键词和向量的混合召回？
- 是否考虑最近性？
- 是否考虑使用次数？
- 是否先做作用域过滤，再做相似度召回？
- 是否在注入前再次做权限过滤？
- 是否限制召回数量？
- 是否避免注入无关记忆？
- 是否记录每次注入？
- 是否能回放模型实际看到的上下文？

### 隐私和治理

- 记忆是否按用户和智能体隔离？
- 记忆是否按租户、项目或组织隔离？
- 用户是否能查看记忆？
- 用户是否能删除记忆？
- 是否有按用户删除的接口？
- 记忆访问是否有审计？
- 是否支持过期、衰减和上限清理？
- 是否避免把原始敏感信息写入长期记忆？

### 轨迹和可观测性

- 是否记录每次工具调用？
- 是否记录每次记忆写入、召回和注入？
- 是否能区分答案来自记忆、工具结果、项目上下文还是模型推理？
- 是否能回放失败任务的关键上下文？
- 是否能统计哪些记忆从未被召回？

### 评测

- 是否有固定召回测试集？
- 是否每次部署都运行？
- Recall@1 和 Recall@K 是否达标？
- 召回回归是否阻塞发布？
- 是否有关闭记忆后的基线结果？
- 是否测试跨用户、跨项目、跨租户的隔离失败场景？

## 结语

智能体的记忆，不是为了让模型“显得更有人味”，而是为了让系统在长任务、跨会话和个性化场景中保持连续性。

但这份连续性有代价。

你不能只问“能不能记住”，还要问：

该不该记？
记多久？
谁能看到？
什么时候召回？
错了怎么纠正？
用户怎么删除？
上线后怎么评测？

我现在更倾向于这样理解智能体记忆：

短期记忆是上下文管理。
长期记忆是状态管理。
召回是检索系统。
注入是提示词组装。
删除和审计是产品责任。
评测是上线门禁。

当我们用这个视角看记忆，它就不再是框架里的一个参数，而是智能体从演示走向生产时必须认真设计的核心子系统。
