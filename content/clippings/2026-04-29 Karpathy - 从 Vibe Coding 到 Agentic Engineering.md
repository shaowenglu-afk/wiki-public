## 中文要点提炼

### 1. 作为程序员感觉落后

新工具迭代太快，每周都有新方法、新工具、新最佳实践，连 Karpathy 自己都跟不上节奏。这种"**落后感**"对所有人都是新常态。

### 2. Software 3.0

- **Software 1.0** = 程序员手写代码
- **Software 2.0** = 训练神经网络权重
- **Software 3.0** = **用自然语言对 LLM 编程**，prompt 就是代码

### 3. Agent 作为 installer

把 Agent 当成软件世界的"**安装程序**"——它进入环境、读取上下文、安装能力、执行任务。

### 4. Menu Gen vs Raw Prompts

直接和原始 LLM 对话（raw prompts）很自由但也容易失控。"**菜单生成式**"（menu gen）的方式——给用户预设选项 + 限定参数——更工程化、更可控。这是从"和模型聊天"到"和工程系统对话"的转变。

### 5. 可验证性与参差能力（jagged skills）

LLM 的能力呈"**锯齿状**"——某些任务超人、某些任务连小学生都不如，且边界难以预测。这要求工程系统必须配备**强可验证性机制**，否则任何 Agent 系统都会在锯齿的低洼处崩溃。

### 6. 从 Vibe Coding 到 Agentic Engineering

> "Vibe coding 是 prompt 一下试运气，Agentic Engineering 是把这个'试'变成系统。"

**Agentic Engineering** = 把 prompt 工程、harness 设计、工具集、评估、反馈循环、专业角色分工，全部组织成**可重复、可迭代、可演化的工程实践**。

### 7. LLM 不是动物，是幽灵

不要拟人化 LLM——它不像狗、不像猫、不像人。**它是"幽灵"——是从海量人类文字中被召唤出来的概率性实体**。需要新的思考方式来对待它：不是训练动物，而是引导一种**形态不定的统计存在**。

### 8. 外包思考 vs 外包理解

> "**You can outsource your thinking but never your understanding.**"
>
> 你可以把思考外包给 Agent，但**永远无法把"理解"外包出去**——理解必须自己保留，否则就失去了判断 Agent 输出对错的能力。
